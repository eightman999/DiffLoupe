# DiffLoupe 比較アルゴリズム仕様書(Swift macOSネイティブ版 v1)

本書は、既存実装である C++/Qt 版([src/core/comparator.cpp](../src/core/comparator.cpp) /
[comparator.h](../src/core/comparator.h) / [diffresult.h](../src/core/diffresult.h))と
Python/PySide6 版([Python/comparator.py](../Python/comparator.py))のコード読解に基づき、
比較アルゴリズムの現行仕様を明文化し、Swift 版 v1 で採用する仕様を確定するものである。

読解対象コミット: `32c49dc`(master)

---

## 1. 既存実装の仕様

### 1.1 ディレクトリ走査

| 項目 | C++版 | Python版 |
|---|---|---|
| 走査方法 | `QDirIterator`(`Files \| Dirs \| NoDotAndDotDot`, `Subdirectories`)で再帰列挙(comparator.cpp:84-85) | `os.walk()` で再帰列挙(comparator.py:44) |
| 再帰深さ | 無制限 | 無制限 |
| ディレクトリのシンボリックリンク | 辿らない(`FollowSymlinks` 未指定) | 辿らない(`followlinks=False` 既定) |
| ファイルのシンボリックリンク | 比較対象。内容はリンク先を読む | 同左 |
| 隠しファイル(`.`始まり) | 既定で除外。`showHidden` で表示可(comparator.cpp:255-258) | 既定で除外。`show_hidden` で表示可(comparator.py:51) |
| 隠しディレクトリ | 除外を**意図**しているが不具合あり(後述 1.4) | サブツリーごと正しく除外(`dirs[:]` プルーニング、comparator.py:46-47) |
| 除外パターン | なし(隠しファイル規則のみ) | なし(同左) |
| ディレクトリ自体のエントリ化 | されない。ファイルのみ相対パスをキーに記録(comparator.cpp:90-109)。**空ディレクトリは結果に現れない** | 同左(comparator.py:49-58) |
| 対象フォルダ数 | N ≧ 2 に対応 | 同左 |
| エントリの照合キー | 基準フォルダからの相対パス(文字列完全一致) | 同左 |
| 結果の並び順 | `std::map` のキー順(QString の辞書順) | `sorted()`(辞書順、comparator.py:63) |

補足: `.git` と `.DS_Store` は「`.`始まり」規則により既定で除外されるが、
`node_modules` はどちらの版でも除外されない。

### 1.2 ファイル同一性の判定基準

両版とも同じ**2段階判定**である(C++版は `useMetadataFirst`(既定 true)で1段目を省略可)。

1. **突き合わせ**: 相対パスをキーに各フォルダのファイルを束ねる。
2. **片側のみ存在**: 内容比較を行わず「欠落」扱い。
   - C++版: `MissingInA` / `MissingInB`(comparator.cpp:140-154、diffresult.h:35-36)
   - Python版: ステータス算出されず、後段で空ハッシュ集合により「一致」となる**不具合**あり(後述 1.4)
3. **メタデータ一次比較**: **サイズが完全一致 かつ mtime が完全一致** → 「一致(メタデータ)」と判定し、**ハッシュ計算を行わない**(comparator.cpp:128-131、comparator.py:72-75)。
   - C++版は `QDateTime` の等値比較、Python版は `st_mtime`(float)の等値比較。
4. **ハッシュ比較**: メタデータ不一致の場合のみ、対象ファイルの **SHA-256** を
   **64KiB チャンクのストリーミング読み込み**で並列計算(C++: `QThreadPool`+`QtConcurrent`、
   Python: `multiprocessing.Pool`)。全ファイルのハッシュ値が1種類以下なら「一致」、
   2種類以上なら「差異」(comparator.cpp:238-247、comparator.py:105-116)。

**バイト単位の直接比較はどちらの版も行わない。**
タイムスタンプのみが異なる同内容ファイルはハッシュ比較により「一致」になる。
逆に「サイズと mtime が一致するが内容が異なる」ファイルは**誤って「一致(メタデータ)」と判定される**
(両版共通の設計上のリスク)。

### 1.3 進捗通知

- C++版: フェーズ単位の固定パーセンテージ(収集中0% → ハッシュ30% → 整理90% → 完了100%)+
  ハッシュ計算の件数進捗(n/m)をコールバック通知(comparator.cpp:37-75, 202-209)。
- Python版: comparator 自体は進捗通知を持たない(GUI 側ワーカーが管理)。
- キャンセル機構はどちらの comparator にもない。

### 1.4 既知の不具合(読解で確認したもの)

| # | 内容 | 該当箇所 |
|---|---|---|
| B1 | C++版: 隠しディレクトリのスキップに `it.next()` を誤用。`QDirIterator` はサブツリーの枝刈りができないため、**隠しディレクトリ配下のファイルは(ファイル名が`.`始まりでない限り)比較対象に含まれてしまう**。さらに直後のエントリを1件余分に読み飛ばすため、無関係なファイルが欠落しうる | comparator.cpp:90-94 |
| B2 | Python版: 片側のみ存在のファイルがハッシュ対象から外れ(comparator.py:69-70)、最終判定で `unique_hashes` が空集合 → `len() <= 1` により**ステータスが「一致」になる**。GUI 側がパスの有無で欠落表示を補正している | comparator.py:102-116 |
| B3 | 両版: 読み込み不可ファイルのハッシュは空値(C++: 空文字列 / Python: None)となり集合から除外されるため、**両側とも読込不可の場合「一致」と誤判定される** | comparator.cpp:238-246、comparator.py:105-116 |

### 1.5 C++版と Python版の相違点まとめ(両論併記)

| 項目 | C++版 | Python版 | Swift v1 での採用案 |
|---|---|---|---|
| 隠しディレクトリ除外 | 機能していない(B1) | サブツリーごと除外 | 除外はパターン方式に変更(§2.2)。プルーニングは Python 版方式(サブツリーごと) |
| 片側のみのステータス | `MissingInA`/`MissingInB` を明示 | 「一致」になる(B2) | C++版方式を採用し `leftOnly`/`rightOnly` を明示 |
| 読込エラーの扱い | 「一致」誤判定(B3) | 同左(B3) | 新設の `error` ステータスで明示(§2.3) |
| 一次比較 | サイズ+mtime(設定で無効化可) | サイズ+mtime(固定) | **mtime を判定に使わない**(§2.3、意図的変更) |
| 進捗・キャンセル | 進捗のみ | なし | 進捗+キャンセル両対応(§2.5) |

## 2. Swift 版 v1 の採用仕様

### 2.1 前提

- 比較対象は**左右2フォルダ**に限定する(既存は N≧2 だが、v1 の製品方針は2ペイン比較)。
- フォルダは `NSOpenPanel` 経由でのみ指定する。
- 実装は Foundation / CryptoKit / Swift Concurrency のみを使用(外部依存ゼロ)。

### 2.2 ディレクトリ走査

- 各フォルダを再帰走査する。深さ無制限。
- **ディレクトリのシンボリックリンクは辿らず、エントリとしても記録しない**(既存両版と同じ)。
- **ファイルのシンボリックリンクは比較対象とし、内容はリンク先を読む**(既存両版と同じ)。
  リンク切れは読込エラーとして `error` 扱い。
- **隠しファイル・隠しディレクトリは比較対象に含める**(既存既定からの変更)。
  - 理由: 本製品の用途(納品物・バックアップの突き合わせ)では全ファイルの把握が重要。
    ノイズは下記の既定除外パターンで抑制する。既存の `showHidden` トグルは採用しない。
- **除外パターン**: 名前が `.git` / `.DS_Store` / `node_modules` に完全一致する
  エントリを既定で除外する(ディレクトリの場合はサブツリーごと除外)。
  設定でこの既定除外を一括オン/オフできる(v1 ではパターン編集は不可)。
- **ディレクトリもエントリとして記録する**(既存からの拡張)。
  これにより空ディレクトリや片側のみのディレクトリも「左のみ/右のみ」としてツリーに表示される。
- 照合キーは相対パス。**NFC 正規化した上で照合する**(既存は未対応。macOS の NFD と
  外部由来 NFC の混在で同名ファイルが別エントリになる事故を防ぐ)。実ファイルアクセスには
  各側で列挙時に得た元の URL を用いる。
- 大文字小文字はファイルシステムが返した名前をそのまま用いる(大小のみ異なる名前は別エントリ)。

### 2.3 ファイル同一性の判定(段階判定)

相対パスで突き合わせた各エントリについて、次の順で判定する。

1. **片側のみ存在** → `leftOnly` / `rightOnly`(内容比較なし)。
2. **種別不一致**(片方ファイル・片方ディレクトリ)→ `different`。
3. **両側ファイル・サイズ不一致** → `different` を確定(ハッシュ計算を省略)。
4. **両側ファイル・サイズ一致** → 両ファイルの **SHA-256(64KiB チャンクのストリーミング)** を計算し、
   - ハッシュ一致 → `identical`
   - ハッシュ不一致 → `different`
5. いずれかの段階で読み込み・stat に失敗 → `error`(B3 の修正)。

**既存からの意図的変更(重要)**: 既存両版の「サイズ+mtime 一致 → 同一(ハッシュ省略)」は
内容相違の見逃しリスクがあるため、**v1 では mtime を判定に一切使わない**。
- サイズ不一致 → 内容は必ず異なるため、ハッシュ省略は安全(高速化)。
- サイズ一致 → 必ずハッシュで確認する(確実性優先)。
- 性能は「サイズ不一致の早期確定」+「ハッシュ計算の並列化」(§2.5)で担保する。
  メタデータ高速モードの再導入は v2 以降の検討事項として WORK_LOG に記録する。

ハッシュアルゴリズムは既存両版と同じ SHA-256(CryptoKit)。チャンクサイズも同じ 64KiB。

### 2.4 分類と表示

ファイルの分類は次の5種:

| ステータス | 意味 | 由来 |
|---|---|---|
| `identical` | 内容一致(ハッシュ一致) | 既存の「一致」 |
| `different` | 内容相違 / 種別不一致 | 既存の「差異」 |
| `leftOnly` | 左フォルダのみに存在 | 既存の `MissingInB` |
| `rightOnly` | 右フォルダのみに存在 | 既存の `MissingInA` |
| `error` | 読込不可・stat 失敗 | 新設 |

ディレクトリの表示ステータスは子孫の集約とする:
- 片側のみに存在するディレクトリ → `leftOnly` / `rightOnly`
- 両側に存在し、全子孫が `identical` → `identical`
- それ以外(子孫に相違・片側のみ・エラーを含む)→ `different`(=「差分を含む」)

ツリーの同一階層の並び順は**ディレクトリ優先、名前昇順**(Finder 互換の
`localizedStandardCompare`)。

### 2.5 並列化・進捗・キャンセル

- 走査とハッシュ計算はメインスレッド外で実行する(Swift Concurrency)。
- ハッシュ計算は `TaskGroup` で並列化し、同時実行数は `min(8, 論理コア数)` に制限する。
- 進捗は2フェーズで通知する: 走査(発見エントリ数)、内容比較(完了 n / 対象 m)。
- ユーザーはいつでもキャンセルできる(`Task` キャンセルを各ループで検査し、
  `CancellationError` で中断する)。既存にはない機能。

### 2.6 テキスト差分表示(v1 新規仕様)

フォルダ比較とは独立した、選択ファイルの表示用仕様。

- **行単位**の差分。`CollectionDifference`(Foundation 標準、Myers 法)で算出する。
- 表示は横並び2ペイン。行の対応付けは、連続する削除ブロックと挿入ブロックを
  index 順にペアリングして「変更行」とし、余りを純粋な「削除行」「追加行」とする。
  配色: 追加=緑系、削除=赤系、変更=黄系ハイライト。
- 文字コードは **UTF-8 固定**。先頭 8KiB に NUL バイトを含む、または UTF-8 として
  デコードできないファイルは「バイナリファイル」と表示し、差分表示しない。
- 行分割は LF 基準とし、**表示・行比較時のみ行末の CR を除去する**(CRLF ファイル対策)。
  注意: フォルダ比較(§2.3)はバイト厳密なので、改行コードのみが異なるファイルは
  「内容相違」に分類される。これは仕様である(差分ビューでは行内容の違いとして見えない場合がある)。
- 巨大ファイル保護: 左右合計 40,000 行を超える場合は差分計算を行わず、その旨を表示する
  (Myers 法は相違が大きいときに計算量が増大するため)。また 32MiB を超えるファイルは
  読み込みを行わず「大きすぎる」と表示する(メモリ保護)。

### 2.7 既知の制限・v2 以降への持ち越し

- メタデータ(mtime)による高速一致モードのオプション化。
- UTF-8 以外のエンコーディング対応(既存 Python 版は chardet による自動判定を持つ)。
- 除外パターンのユーザー編集。
- ハードリンク・FIFO 等の特殊ファイルは対象外(走査時に通常ファイル・ディレクトリ以外はスキップ)。
- 同一ボリューム上の APFS クローンの高速判定(clonefile 検出)は行わない。
